# 兑换记录后台管理系统 — 优化提示词

## 项目背景

基于现有暑假作业打卡系统（FastAPI + Vue3 CDN + SQLite），开发**兑换记录后台管理模块**，供管理员对用户提交的兑换申请进行审核与处理。

---

## 一、数据模型设计

### 1.1 Redemption 表扩展字段

在现有 `Redemption` 模型基础上，新增以下字段：

```python
# 已有字段
id = Column(Integer, primary_key=True)
user_id = Column(Integer, ForeignKey("users.id"), nullable=False)
prize_id = Column(Integer, ForeignKey("prizes.id"), nullable=False)
prize_name = Column(String(128), nullable=False)
cost_points = Column(Integer, default=0)
redeemed_at = Column(DateTime, default=datetime.utcnow)
status = Column(String(16), default="pending")  # pending|approved|rejected
note = Column(String(256), nullable=True)

# 新增字段
review_note = Column(String(256), nullable=True)      # 审核备注
reviewed_by = Column(Integer, ForeignKey("users.id"), nullable=True)  # 操作管理员
reviewed_at = Column(DateTime, nullable=True)          # 操作时间
```

### 1.2 状态定义

| 状态值 | 显示名称 | 含义 |
|--------|---------|------|
| `pending` | 待核实 | 用户已提交兑换申请，等待管理员审核 |
| `approved` | 已兑现 | 管理员核实通过，奖品已发放 |
| `rejected` | 已拒绝 | 管理员核实不通过，拒绝发放 |

## 二、后端 API 设计

### 2.1 兑换记录列表

```
GET /api/admin/redemptions
```

**查询参数：**
- `status`（可选）：按状态过滤，值为 `pending` / `approved` / `rejected`
- `page`（可选，默认 1）：页码
- `page_size`（可选，默认 20）：每页条数

**响应示例：**
```json
{
  "items": [
    {
      "id": 1,
      "user_id": 3,
      "nickname": "小明",
      "prize_name": "精美笔记本",
      "cost_points": 50,
      "redeemed_at": "2026-07-07 14:30",
      "status": "pending",
      "note": "用户备注",
      "review_note": null,
      "reviewed_by": null,
      "reviewed_at": null
    }
  ],
  "total": 25,
  "page": 1,
  "page_size": 20
}
```

### 2.2 兑换记录详情

```
GET /api/admin/redemptions/{id}
```

**响应示例：**
```json
{
  "id": 1,
  "user_id": 3,
  "nickname": "小明",
  "username": "xiaoming",
  "prize_name": "精美笔记本",
  "prize_description": "A5 尺寸，100 页",
  "cost_points": 50,
  "redeemed_at": "2026-07-07 14:30",
  "status": "pending",
  "note": "用户备注",
  "review_note": null,
  "reviewed_by": null,
  "reviewed_at": null
}
```

### 2.3 审核操作

```
PUT /api/admin/redemptions/{id}/review
```

**请求体：**
```json
{
  "action": "approve",  // "approve" 或 "reject"
  "note": "核实通过，奖品已发放"  // 可选，审核备注
}
```

**响应示例：**
```json
{
  "ok": true,
  "message": "已兑现",
  "reviewed_at": "2026-07-07 15:00",
  "reviewed_by": "admin"
}
```

**错误处理：**
- 404：记录不存在
- 400：记录已审核，不可重复操作
- 403：无管理员权限

### 2.4 统计数据

```
GET /api/admin/stats
```

在现有统计接口中新增：
```json
{
  "redeem_pending": 5,
  "redeem_approved": 12,
  "redeem_rejected": 3
}
```

## 五、安全性要求

### 5.1 JWT 鉴权

所有管理接口需验证：
```
Authorization: Bearer <token>
```

### 5.2 角色校验

接口层校验 `user.role == "admin"`，非管理员返回 403：
```python
@router.put("/redemptions/{id}/review")
def review_redeem(id: int, body: ReviewBody, user: User = Depends(require_role("admin")), db: Session = Depends(get_db)):
    ...
```

### 5.3 防重复操作

后端校验 `status != "pending"` 时拒绝操作：
```python
if redeem.status != "pending":
    raise HTTPException(status_code=400, detail="该记录已审核，不可重复操作")
```

### 5.4 操作日志

每次审核操作记录：
- `reviewed_by`：操作管理员 ID
- `reviewed_at`：操作时间
- `review_note`：审核备注

### 5.5 前端守卫

- 未登录管理账号时跳转登录页
- 路由级别拦截非管理员访问

## 九、示例代码片段

### 9.1 后端审核接口

```python
class ReviewBody(BaseModel):
    action: str  # "approve" | "reject"
    note: str | None = None

@router.put("/redemptions/{id}/review")
def review_redeem(
    id: int,
    body: ReviewBody,
    user: User = Depends(require_role("admin")),
    db: Session = Depends(get_db)
):
    redeem = db.query(Redemption).filter(Redemption.id == id).first()
    if not redeem:
        raise HTTPException(status_code=404, detail="记录不存在")
    if redeem.status != "pending":
        raise HTTPException(status_code=400, detail="该记录已审核，不可重复操作")
    
    redeem.status = "approved" if body.action == "approve" else "rejected"
    redeem.review_note = body.note or ""
    redeem.reviewed_by = user.id
    redeem.reviewed_at = datetime.utcnow()
    db.commit()
    
    return {
        "ok": True,
        "message": "已兑现" if body.action == "approve" else "已拒绝",
        "reviewed_at": redeem.reviewed_at.strftime("%Y-%m-%d %H:%M"),
        "reviewed_by": user.nickname
    }
```

### 9.2 前端审核方法

```javascript
async submitReview() {
  const { action, note, redeemId } = this.reviewing;
  try {
    await this.api(`/api/admin/redemptions/${redeemId}/review`, {
      method: "PUT",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ action, note }),
    });
    this.showToast(action === "approve" ? "已兑现" : "已拒绝");
    this.reviewing = null;
    await this.loadRedeems();
    this.stats = await this.api("/api/admin/stats");
  } catch (e) {
    this.showToast(e.message || "操作失败");
  }
}
```

## 十、注意事项

1. **数据库迁移**：新增字段需确保兼容现有数据，建议设置 `nullable=True` 或提供默认值
2. **分页性能**：数据量大时考虑为 `status`、`user_id` 添加索引
3. **时间格式**：统一使用 UTC 存储，前端展示时转换为本地时间
4. **错误提示**：操作失败时给出明确、用户友好的错误信息
5. **响应式测试**：在 Chrome DevTools 中模拟不同设备尺寸（iPhone SE / iPhone 12 / iPad）验证布局
