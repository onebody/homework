# 功能开发总结

## 一、家长绑定孩子功能

### 1.1 功能概述

家长使用学生在"我的"页面查看的**绑定码**来绑定孩子账号，绑定后家长可以：
- 代孩子打卡（提交作业照片）
- 查看孩子的打卡连续天数、积分
- 使用孩子的积分在商城兑换奖品
- 查看孩子的学习报告
- 一个家长可绑定多个孩子

### 1.2 绑定码生成机制

- **格式**：`S` + 5位数字ID，如 `S00001`、`S00014`
- **生成时机**：学生注册成功时由后端自动生成
- **存储位置**：`users` 表的 `bind_code` 字段
- **查看方式**：学生登录后在"我的"页面可见绑定码

### 1.3 后端接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/parent/bind` | POST | 家长绑定孩子（需 `child_username` + `bind_code`） |
| `/api/parent/children` | GET | 获取当前家长的已绑定孩子列表 |
| `/api/auth/register` | POST | 注册时自动生成学生的 `bind_code` |

**绑定端流程**：
1. 校验当前用户角色必须为 `parent`
2. 根据 `child_username` 查找学生账号
3. 比对 `bind_code` 是否一致
4. 检查绑定关系是否已存在（幂等性处理）
5. 创建 `StudentParent` 关联记录

**幂等性**：重复绑定同一孩子返回 `{"ok": true, "message": "已绑定"}`，不产生错误。

### 1.4 数据模型

```python
class StudentParent(Base):
    __tablename__ = "student_parents"
    id: int (PK)
    student_id: int (FK → users.id)
    parent_id: int (FK → users.id)
    created_at: datetime
```

`User` 模型新增字段：
```python
bind_code: str | None  # 绑定码，如 "S00001"
```

### 1.5 前端集成

- **家长端首页（无孩子时）**：显示绑定面板，包含孩子用户名和绑定码两个输入框
- **家长端"我的"页面**：显示"绑定更多孩子"面板，支持绑定多个孩子
- **绑定成功后**：自动刷新孩子列表，导航到首页

## 二、双角色支持（学生端 / 家长端）

### 2.1 角色定义

| 角色 | 标识 | 功能范围 |
|------|------|----------|
| 学生 | `student` | 打卡、查看积分、兑换奖品、抽奖、闯关任务 |
| 家长 | `parent` | 绑定孩子、代打卡、代兑换、查看报告 |
| 管理员 | `admin` | 审核打卡、管理奖品、管理系统（管理后台） |

### 2.2 前端双角色复用

学生端 H5 页面同时服务于学生和家长两种角色：
- 同一页面根据 `user.role` 自动切换界面
- 家长登录后显示"孩子切换"导航栏
- 无孩子时显示绑定面板
- 所有操作（打卡、兑换、抽奖）均通过 `actingChildId` 确定操作主体

### 2.3 权限控制

后端通过 `deps.py` 的 `get_current_user` 和 `require_role` 实现 RBAC：
- 学生接口：仅 `student` 角色可访问
- 家长接口：仅 `parent` 角色可访问
- 管理接口：仅 `admin` 角色可访问

## 三、数据库迁移机制（Alembic）

### 3.1 架构概览

```
backend/
├── alembic/
│   ├── env.py              # 迁移环境配置
│   ├── script.py.mako      # 迁移脚本模板
│   └── versions/
│       └── 001_initial.py  # 初始迁移脚本（9张表）
├── alembic.ini             # Alembic 配置文件
├── migrate.py              # 自动迁移入口
└── seed.py                 # 种子数据初始化
```

### 3.2 三种部署场景

`migrate.py` 自动处理三种部署场景：

| 场景 | 数据库状态 | 处理逻辑 |
|------|-----------|----------|
| **首次部署** | 数据库不存在或无表 | `create_all` 建表 + stamp 初始版本 |
| **已有数据无版本** | 表存在但 `alembic_version` 为空 | stamp 到初始版本，不修改表结构 |
| **正常增量迁移** | 表存在且有版本记录 | `alembic upgrade head` 执行迁移 |

### 3.3 迁移流程

```
migrate.py 启动 → 备份数据库 → 执行迁移 → 回填绑定码 → 执行种子数据
```

### 3.4 代码变更流程

当需要修改数据库 schema 时：

```bash
# 1. 修改 models.py
# 2. 生成迁移脚本
alembic revision --autogenerate -m "描述"

# 3. 执行增量部署
./scripts/deploy.sh local   # 本地
./scripts/deploy.sh prod    # 生产
```

## 四、安全加固措施

### 4.1 移除明文密码

| 位置 | 原内容 | 修改后 |
|------|--------|--------|
| 管理端登录页 | `默认账号 admin / admin123` | `请联系系统管理员获取账号` |
| `seed.py` | 硬编码 `admin123` | `ADMIN_INIT_PASSWORD` 环境变量或随机生成 |
| `config.py` | 硬编码 SECRET | `.secret_key` 文件或 `SUMMER_SECRET` 环境变量 |
| `docker-compose.yml` | 硬编码 SECRET | `${SUMMER_SECRET:-fallback}` 变量引用 |
| `README.md` | 明文账号 | 已移除 |

### 4.2 当前密码策略

- **管理员密码**：通过 `ADMIN_INIT_PASSWORD` 环境变量设置，未设置则随机生成并打印
- **JWT 密钥**：通过 `.secret_key` 文件持久化，或 `SUMMER_SECRET` 环境变量
- **用户密码**：PBKDF2 哈希 + 随机盐存储
- **Git 安全**：`.gitignore` 排除 `.secret_key`、`*.db`、`.env`、`.dockerignore` 排除 `.secret_key`

### 4.3 签权机制

- JWT Token（包含 `uid` + `role` + `exp`）
- Bearer Token 鉴权
- 装饰器级别权限控制（`require_role`）

## 五、增量部署机制

### 5.1 部署脚本

`scripts/deploy.sh` 支持两种模式：

```bash
./deploy.sh local   # 本地 Docker 增量更新
./deploy.sh prod    # 生产服务器增量更新
```

### 5.2 部署流程

```
备份数据库 → 传输代码（生产） → 构建镜像 → 停止旧容器 → 
删除旧容器（保留数据卷） → 启动新容器 → 验证健康检查
```

### 5.3 关键原则

- **严禁直接删除数据库文件**
- 使用 Docker 数据卷持久化数据
- 迁移脚本（`migrate.py`）自动处理 schema 变更
- 种子数据幂等（检查已存在则跳过）
- 部署前自动备份数据库到 `backups/` 目录

## 七、系统架构

### 7.1 技术栈

**后端**：
- **语言**：Python 3.11+
- **框架**：FastAPI（异步 Web 框架）
- **ORM**：SQLAlchemy 2.0（异步兼容）
- **数据库**：SQLite（SQLite WAL 模式支持并发读写）
- **认证**：JWT（PyJWT）+ PBKDF2 密码哈希
- **迁移**：Alembic 1.13.2

**前端**：
- **框架**：Vue.js 3（CDN 加载）
- **样式**：原生 CSS（变量驱动）
- **部署**：FastAPI StaticFiles 服务静态资源

**部署**：
- **容器化**：Docker + docker-compose
- **生产地址**：`192.168.1.112:6565`
- **本地地址**：`localhost:8000`
