## 打卡积分 + 积分兑换奖品 通用模块（points-system/）— 已新建并验证

- 独立于 `summer-homework-checkin`，是「打卡赚积分→积分兑换奖品」逻辑的**干净最小化实现**，便于移植/复用。
- 技术栈：FastAPI 0.137 + SQLAlchemy 2.0.50 + SQLite + 原生 HTML/JS（无 CDN 依赖）。受管 venv `default` 已含全部依赖。
- 6 张表：`users / point_accounts / point_ledgers / checkins / prizes / redemptions`。核心服务 `backend/app/services/points_service.py`（事务化 do_checkin / do_redeem）。
- 业务规则：`POINTS_PER_CHECKIN=10`，连续 7 天额外 `POINTS_STREAK_BONUS=20`；兑换需校验 有效期/库存/余额，同事务内扣余额+扣库存+写流水+生成兑换记录。
- 数据一致性：单 Session 事务 commit/rollback；`checkins(user_id, check_date)` 唯一约束兜底防重复打卡；Postgres 升级路径用 `with_for_update()` 悲观锁。
- 演示服务运行在 **http://127.0.0.1:8011**（避开既有项目 8000 端口）；前端首页 `/`，API 前缀 `/api`，演示账号 小明/小红。
- 端到端测试 24/24 通过（连续奖励、防重复、积分不足/库存不足拦截、兑换后余额库存同步、流水对账）。
- 踩坑记录：① SQLAlchemy SQLite 的 `DateTime(timezone=True)` 读取仍返回 naive，须配合 naive UTC 的 `now` 比较（用 `datetime.now(timezone.utc).replace(tzinfo=None)`）；② `Response` 模型字段必须显式构造（如 `RedemptionOut.prize_name` 取自 `prize.name`，不能直接返回 ORM 对象）。

- 踩坑/关键修复：**并发「丢失更新」**——默认 SQLite DEFERRED 事务下，两个并发 convert 读到同一余额各扣 50、后者覆盖前者（40 并发仅 4 生效）。`isolation_level="IMMEDIATE"` 在 SQLAlchemy+SQLite 非法（仅 READ UNCOMMITTED/SERIALIZABLE/AUTOCOMMIT）。最终方案：写路径 `do_convert/do_draw` 加进程内 `threading.Lock` 串行化读-改-写（单进程 uvicorn 有效），README 注明多实例改 Postgres `with_for_update()`。
